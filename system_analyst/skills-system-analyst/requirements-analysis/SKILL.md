# Requirements Analysis

> **Meta:** v1.0.0 | 23-02-2026

## Назначение

Навык для сбора, анализа и документирования требований к системе. Включает выявление функциональных и нефункциональных требований, создание пользовательских историй, определение критериев приёмки и формирование документации требований для передачи команде разработки и Project Manager.

## Когда использовать

Используйте этот навык:
- На этапе инициации и планирования проекта
- При сборе требований от заинтересованных сторон
- При формализации функциональных и нефункциональных требований
- При создании пользовательских историй и сценариев использования
- Для подготовки данных для Project Manager (оценки, риски)

## Функции

### Requirements Gathering
Сбор требований от различных источников:
- Интервью с заинтересованными сторонами
- Анализ бизнес-процессов (BPMN)
- Анализ существующих систем
- Изучение документов и спецификаций
- Мозговой штурм с командой

### Requirements Categorization
Классификация требований по типам:
- Функциональные требования (что система должна делать)
- Нефункциональные требования (как система должна работать)
  - Производительность
  - Безопасность
  - Масштабируемость
  - Надёжность
  - Юзабилити
- Интерфейсные требования
- Ограничения (технические, бизнес, законодательные)
- Предположения

### Requirements Prioritization
Приоритизация требований по критериям:
- Влияние на бизнес-цели
- Критичность для пользователей
- Сложность реализации
- Риски и зависимости
- MoSCoW (Must have, Should have, Could have, Won't have)

### Requirements Documentation
Документирование требований в структурированном виде:
- Создание SRS (Software Requirements Specification)
- Описание пользовательских историй (User Stories)
- Формат: "As a <role>, I want <feature>, so that <benefit>"
- Создание acceptance criteria (Gherkin: Given-When-Then)
- Диаграммы Use Case (UML)

## Интеграция с Project Manager

### Данные для Project Manager
Предоставляет следующие данные для PM:
- **Оценки трудозатрат:** предварительные оценки сложности требований
- **Риски требований:** требования с высокой неопределённостью
- **Зависимости:** связи между требованиями и внешними системами
- **Приоритеты:** обоснование приоритизации требований

### Взаимодействие
- PM запрашивает анализ требований для планирования
- PM получает данные для оценки бюджета и сроков
- PM использует приоритеты для планирования итераций
- SA валидирует изменения требований с PM

## Примеры использования

### Пример 1: Анализ требований для CRM-системы

**Функциональные требования:**
| ID | Требование | Приоритет | Оценка |
|----|------------|-----------|--------|
| FR-001 | Управление контактами клиентов | Must | 40 ч |
| FR-002 | Отслеживание продаж | Must | 60 ч |
| FR-003 | Управление задачами | Should | 30 ч |
| FR-004 | Генерация отчётов | Could | 20 ч |

**Нефункциональные требования:**
| ID | Требование | Метрика |
|----|------------|---------|
| NFR-001 | Производительность | Время отклика < 2 сек |
| NFR-002 | Масштабируемость | 1000+ одновременных пользователей |
| NFR-003 | Доступность | 99.9% uptime |
| NFR-004 | Безопасность | Соответствие GDPR |

**Пользовательские истории:**
```
US-001: Управление контактами
As a менеджер по продажам,
I want создавать, редактировать и удалять контакты клиентов,
So that я могу вести базу клиентов.

Acceptance Criteria:
Given я нахожусь на странице контактов
When я нажимаю "Добавить контакт"
Then открывается форма создания контакта
And все обязательные поля отмечены asterisk
```

### Пример 2: Анализ требований для мобильного приложения

**Функциональные требования:**
- Регистрация и авторизация пользователей
- Поиск и бронирование услуг
- Платежи и подтверждение
- Отзывы и рейтинги

**Нефункциональные требования:**
- Работа на iOS 12+ и Android 8+
- Поддержка оффлайн-режима
- Время запуска менее 3 секунд
- Энергоэффективность

**Данные для PM:**
- Общая оценка трудозатрат: 320 часов
- Высокие риски: интеграция платёжной системы
- Критические зависимости: API сторонних сервисов
