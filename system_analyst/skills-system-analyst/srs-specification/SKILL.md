# Software Requirements Specification (SRS)

> **Meta:** v1.0.0 | 23-02-2026

## Назначение

Навык для создания детализированной спецификации требований к программному обеспечению (SRS - Software Requirements Specification). Включает формализацию функциональных и нефункциональных требований, определение ограничений и допущений, создание структурированного документа SRS по стандарту IEEE 830, который служит основой для проектирования, разработки и тестирования.

## Когда использовать

Используйте этот навык:
- При подготовке детализированной спецификации требований
- Когда нужно формализовать и структурировать собранные требования
- Для создания документа, который будет использоваться командой разработки
- При необходимости согласования требований с заинтересованными сторонами
- Для передачи Project Manager данных для планирования

## Функции

### Functional Requirements Specification
Определение функциональных требований:
- Описание функций системы
- Входные и выходные данные
- Условия выполнения функций
- Взаимодействие с пользователями
- Интеграция с другими системами
- Бизнес-правила и валидация

### Non-functional Requirements Specification
Определение нефункциональных требований:
- **Производительность:** время отклика, пропускная способность
- **Масштабируемость:** горизонтальное и вертикальное масштабирование
- **Безопасность:** аутентификация, авторизация, защита данных
- **Надёжность:** доступность, отказоустойчивость, восстановление
- **Юзабилити:** пользовательский опыт, доступность (accessibility)
- **Совместимость:** браузеры, устройства, операционные системы
- **Поддерживаемость:** логирование, мониторинг, отладка

### Constraints and Assumptions
Формулирование ограничений и допущений:
- Технические ограничения (технологии, платформы)
- Ограничения по времени и бюджету
- Законодательные и нормативные требования
- Допущения об окружающей среде
- Ограничения внешних систем
- Ограничения ресурсов команды

### SRS Document Creation
Создание документа SRS по стандарту IEEE 830:
- Структура документа
- Форматирование требований
- Приоритизация требований
- Версионность документа
- Трейсирование требований
- Утверждение и контроль изменений

## Интеграция с Project Manager

### Данные для Project Manager
Предоставляет следующие данные для PM:
- **Объём работ:** количество требований для оценки
- **Сложность:** классификация требований по сложности
- **Риски:** требования с высокой неопределённостью
- **Зависимости:** связи между требованиями
- **Приоритеты:** обоснование для планирования итераций

### Взаимодействие
- PM запрашивает SRS для планирования проекта
- PM получает данные для оценки бюджета и сроков
- PM использует приоритеты для планирования релизов
- SA валидирует изменения требований с PM

## Примеры использования

### Пример 1: SRS для веб-приложения управления задачами

**Функциональные требования:**

| ID | Требование | Описание | Приоритет |
|----|------------|----------|-----------|
| FR-001 | Создание задач | Пользователь может создавать задачи с названием, описанием, сроком | Must |
| FR-002 | Назначение задач | Пользователь может назначать задачи другим пользователям | Must |
| FR-003 | Фильтрация задач | Система предоставляет фильтрацию по статусу, исполнителю, сроку | Should |
| FR-004 | Отчёты | Система генерирует отчёты о выполнении задач | Could |

**Нефункциональные требования:**

| ID | Требование | Метрика | Обоснование |
|----|------------|---------|-------------|
| NFR-001 | Производительность | Время отклика < 2 сек | UX требования |
| NFR-002 | Масштабируемость | 500 одновременных пользователей | Планируемый рост |
| NFR-003 | Доступность | 99.5% uptime | SLA |
| NFR-004 | Безопасность | SSL/TLS для всех соединений | Защита данных |

**Данные для PM:**
- Общее количество требований: 15 функциональных, 8 нефункциональных
- Оценка сложности: 8 простых, 4 средних, 3 сложных
- Риски: интеграция с внешним календарём
- Критические зависимости: API уведомлений

### Пример 2: SRS для мобильного банкинг-приложения

**Функциональные требования:**
- Аутентификация пользователей (биометрия, PIN)
- Просмотр баланса счетов и истории транзакций
- Переводы между счетами и внешние переводы
- Платежи по реквизитам и QR-коду
- Управление картами (блокировка, лимиты)

**Нефункциональные требования:**
- Поддержка iOS 12+ и Android 8+
- Время запуска приложения не более 3 секунд
- Поддержка оффлайн-режима для просмотра истории
- Соответствие стандартам безопасности PCI DSS
- Шифрование данных на устройстве

**Ограничения:**
- Интеграция с банковским ядром через API
- Соответствие требованиям ЦБ РФ
- Локализация данных на территории РФ

**Данные для PM:**
- Общая оценка трудозатрат: 450 часов
- Высокие риски: безопасность, соответствие регуляторам
- Критические зависимости: банковское ядро, SMS-провайдер
