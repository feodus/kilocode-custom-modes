# Requirements Management

> **Meta:** v1.0.0 | 23-02-2026

## Назначение

Навык для управления жизненным циклом требований на протяжении всего проекта. Включает трейсирование требований, контроль изменений, координацию с System Analyst, валидацию полноты требований и управление областью проекта (scope management). НЕ включает анализ и документирование требований - это функция System Analyst.

## Когда использовать

Используйте этот навык:
- При управлении изменениями требований
- Для контроля области проекта (scope)
- При координации с System Analyst по требованиям
- Для валидации полноты требований перед планированием
- При трейсировании требований к задачам и тестам

## Функции

### Requirements Traceability Management
Управление трейсированием требований:
- Создание матрицы трейсирования (RTM - Requirements Traceability Matrix)
- Связывание требований с задачами, тестами, кодом
- Отслеживание покрытия требований
- Выявление orphan требований (без реализации)
- Выявление orphan задач (без требований)

### Requirements Change Control
Контроль изменений требований:
- Процесс запроса изменений (Change Request)
- Оценка влияния изменений на проект
- Анализ влияния на бюджет и сроки
- Согласование изменений с заинтересованными сторонами
- Обновление baseline требований

### Scope Management
Управление областью проекта:
- Определение границ проекта (in-scope / out-of-scope)
- Контроль scope creep (неконтролируемое расширение)
- Приоритизация требований на основе бизнес-целей
- Управление бэклогом требований
- Планирование релизов на основе приоритетов

### Requirements Validation
Валидация требований:
- Проверка полноты требований от System Analyst
- Проверка согласованности требований
- Валидация приоритетов с бизнес-целями
- Проверка реализуемости требований
- Подготовка к approval требований

## Интеграция с System Analyst

### Запрос данных от System Analyst
PM запрашивает у SA следующие данные:
- **SRS документ** - спецификация требований
- **Оценки трудозатрат** - предварительные оценки по требованиям
- **Риски требований** - требования с высокой неопределённостью
- **Зависимости** - связи между требованиями и внешними системами
- **Приоритеты** - обоснование приоритизации

### Предоставление данных System Analyst
PM предоставляет SA:
- Утверждённые изменения требований
- Обновлённые приоритеты от бизнеса
- Обратную связь от команды разработки
- Данные о реализуемости требований

### Процесс координации
1. PM запрашивает SRS у System Analyst
2. PM валидирует полноту и согласованность
3. PM создаёт матрицу трейсирования
4. PM запрашивает уточнения при необходимости
5. PM включает требования в план проекта

## Примеры использования

### Пример 1: Матрица трейсирования требований

| Требование | User Story | Задача | Test Case | Статус |
|------------|------------|--------|-----------|--------|
| FR-001 | US-001 | TASK-101 | TC-001, TC-002 | Done |
| FR-002 | US-002 | TASK-102 | TC-003 | In Progress |
| FR-003 | US-003 | - | - | Not Started |
| FR-004 | - | - | - | Missing |

**Анализ:**
- FR-003 не имеет задачи - требуется создание
- FR-004 не имеет User Story - запросить у SA

### Пример 2: Запрос данных у System Analyst

**Запрос PM:**
```
System Analyst, для планирования спринта #3 необходимы следующие данные:
1. Детализация требований FR-005, FR-006 (User Stories, Acceptance Criteria)
2. Оценка трудозатрат для новых требований
3. Зависимости между FR-005 и существующими требованиями
4. Риски, связанные с реализацией FR-006

Срок предоставления: 25.02.2026
```

**Ответ System Analyst:**
```
Предоставляю запрошенные данные:
1. US-005, US-006 с Acceptance Criteria - во вложении
2. Оценка: FR-005 - 16ч, FR-006 - 24ч
3. Зависимости: FR-005 зависит от FR-002 (блокирующая)
4. Риски: FR-006 требует интеграции с внешним API - высокая неопределённость
```

### Пример 3: Контроль изменений требований

**Change Request:**
| Поле | Значение |
|------|----------|
| ID | CR-001 |
| Требование | FR-003 |
| Запрошенное изменение | Добавить функцию экспорта в PDF |
| Инициатор | Бизнес-заказчик |
| Дата запроса | 20.02.2026 |

**Анализ влияния:**
- Влияние на бюджет: +40 часов ($2,000)
- Влияние на сроки: +1 неделя
- Зависимости: требует библиотеку PDF generation
- Риски: совместимость с существующим функционалом

**Решение:** Отклонено в текущем спринте, перенесено в backlog

## Процесс управления требованиями

### 1. Инициация
1. Запросить SRS у System Analyst
2. Провести валидацию полноты
3. Создать матрицу трейсирования
4. Зафиксировать baseline требований

### 2. Планирование
1. Приоритизировать требования
2. Распределить по итерациям/спринтам
3. Оценить ресурсы на реализацию
4. Согласовать с заинтересованными сторонами

### 3. Исполнение
1. Отслеживать статус требований
2. Управлять изменениями
3. Координировать с System Analyst
4. Обновлять матрицу трейсирования

### 4. Контроль
1. Контролировать scope creep
2. Анализировать покрытие требований
3. Проводить ревью изменений
4. Отчитываться перед стейкхолдерами

### 5. Завершение
1. Верифицировать реализацию всех требований
2. Подтвердить покрытие тестами
3. Получить sign-off от заказчика
4. Зафиксировать финальный статус
