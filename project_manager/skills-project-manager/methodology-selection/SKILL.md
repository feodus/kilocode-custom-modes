# Methodology Selection

> **Meta:** v1.0.0 | 22-02-2026

## Purpose

Skill for selecting appropriate software development methodology based on project characteristics, stakeholder requirements, and implementation context. Helps determine the most effective approach to project management and development.

## When to Use

Use this skill:
- At project initiation stage to determine development approach
- When selecting between different methodologies is needed
- When adapting standard methodologies to project specifics
- When integrating with existing organizational processes
- For explaining chosen approach to stakeholders

## Functions

### Methodology Analysis
Analyzing different methodologies:
- Waterfall (cascade model)
- Agile (flexible methodologies)
- Iterative (iterative model)
- V-Model (V-shaped model)
- Spiral (spiral model)
- DevOps/DevSecOps approaches
- Hybrid approaches

### Project Characteristic Assessment
Assessing project characteristics:
- Project size and complexity
- Requirements certainty level
- Stakeholder involvement level
- Uncertainty risk
- Quality and security requirements
- Time constraints

### Stakeholder Requirements Analysis
Analyzing stakeholder requirements:
- Readiness for changes
- Delivery frequency expectations
- Documentation requirements
- Involvement level in the process
- Communication preferences
- Organizational culture

### Hybrid Approach Design
Creating hybrid approach:
- Combining elements of different methodologies
- Adapting to project context
- Accounting for constraints and opportunities
- Balancing flexibility and control
- Accounting for regulatory requirements

## Integration with System Analyst

- Receiving requirements characteristics information from System Analyst
- Accounting for architectural features when selecting methodology
- Analyzing dependency on external systems
- Assessing integration complexity
- Accounting for technology recommendations

## Usage Examples

### Example 1: Selecting Methodology for Government System Project
Context:
- High security requirements
- Clearly defined functional requirements
- Complex change approval procedure
- Need for complete documentation

Selection: V-Model or Waterfall with Agile elements for development
Reasons:
- Need for strict verification and validation
- High documentation requirements
- Limited flexibility in requirements changes

### Example 2: Selecting Methodology for Startup Project
Context:
- High requirements uncertainty
- Rapid market changes
- Limited budget
- Need for quick feedback

Selection: Pure Agile (Scrum/Kanban)
Reasons:
- Flexibility in requirements changes
- Fast value delivery
- Minimal documentation
- Frequent user feedback

### Example 3: Selecting Methodology for Corporate Project
Context:
- Medium requirements certainty
- Moderate time constraints
- Complex organization with different departments
- Existing project management processes

Selection: Hybrid (Agile for development, Waterfall for project management)
Reasons:
- Balance between flexibility and control
- Compatibility with existing processes
- Accounting for organizational constraints
- Manageability for management
