# Methodology Selection

> **Meta:** v1.0.0 | 22-02-2026

## Назначение

Навык для выбора подходящей методологии разработки программного обеспечения на основе характеристик проекта, требований заинтересованных сторон и контекста реализации. Помогает определить наиболее эффективный подход к управлению проектом и разработке.

## Когда использовать

Используйте этот навык:
- На этапе инициации проекта для определения подхода к разработке
- При необходимости выбора между различными методологиями
- При адаптации стандартных методологий под специфику проекта
- При интеграции с существующими процессами в организации
- Для объяснения выбранного подхода заинтересованным сторонам

## Функции

### Methodology Analysis
Анализ различных методологий:
- Waterfall (каскадная модель)
- Agile (гибкие методологии)
- Iterative (итерационная модель)
- V-Model (V-образная модель)
- Spiral (спиральная модель)
- DevOps/DevSecOps подходы
- Гибридные подходы

### Project Characteristic Assessment
Оценка характеристик проекта:
- Размер и сложность проекта
- Степень определенности требований
- Уровень вовлеченности заинтересованных сторон
- Риск неопределенности
- Требования к качеству и безопасности
- Временные ограничения

### Stakeholder Requirements Analysis
Анализ требований заинтересованных сторон:
- Уровень готовности к изменениям
- Ожидания по частоте доставки
- Требования к документации
- Уровень вовлеченности в процесс
- Предпочтения по коммуникации
- Культура организации

### Hybrid Approach Design
Создание гибридного подхода:
- Комбинирование элементов разных методологий
- Адаптация под контекст проекта
- Учет ограничений и возможностей
- Баланс между гибкостью и контролем
- Учет нормативных требований

## Интеграция с System Analyst

- Получение информации о характеристиках требований от System Analyst
- Учет архитектурных особенностей при выборе методологии
- Анализ зависимости от внешних систем
- Оценка сложности интеграции
- Учет рекомендаций по технологиям

## Примеры использования

### Пример 1: Выбор методологии для проекта государственной системы
Контекст:
- Высокие требования к безопасности
- Четко определенные функциональные требования
- Сложная процедура утверждения изменений
- Необходимость полной документации

Выбор: V-Model или Waterfall с элементами Agile для разработки
Причины:
- Необходимость строгой верификации и валидации
- Высокие требования к документации
- Ограниченная гибкость в изменениях требований

### Пример 2: Выбор методологии для стартап-проекта
Контекст:
- Высокая неопределенность требований
- Быстрые изменения рынка
- Ограниченный бюджет
- Необходимость быстрого получения обратной связи

Выбор: Pure Agile (Scrum/Kanban)
Причины:
- Гибкость в изменениях требований
- Быстрая доставка ценности
- Минимальная документация
- Частая обратная связь от пользователей

### Пример 3: Выбор методологии для корпоративного проекта
Контекст:
- Средняя определенность требований
- Умеренные ограничения по времени
- Сложная организация с разными подразделениями
- Существующие процессы управления проектами

Выбор: Hybrid (Agile в разработке, Waterfall для управления проектом)
Причины:
- Баланс между гибкостью и контролем
- Совместимость с существующими процессами
- Учет организационных ограничений
- Управляемость для руководства
