# Kanban Flow Management

> **Meta:** v1.0.0 | 22-02-2026

## Назначение

Навык для управления потоком задач по Kanban-методологии. Включает настройку и ведение Kanban-досок, установку WIP-лимитов, анализ потоков работы, измерение времени выполнения задач (Lead Time/Cycle Time) и оптимизацию процессов. Помогает визуализировать работу, управлять загрузкой команды и повышать эффективность.

## Когда использовать

Используйте этот навык:
- При управлении проектами с непрерывным потоком задач
- Когда важна визуализация процесса работы
- При необходимости управления загрузкой команды
- Для оптимизации времени выполнения задач
- При работе с командами, которые не готовы к полному переходу на Scrum

## Функции

### Kanban Board Setup
Настройка Kanban-досок:
- Определение колонок процесса
- Настройка правил перехода задач
- Установка политик работы с задачами
- Настройка автоматизации
- Интеграция с инструментами отслеживания

### WIP Limit Management
Управление WIP-лимитами:
- Определение оптимальных лимитов
- Мониторинг соблюдения лимитов
- Корректировка лимитов по мере необходимости
- Учет особенностей команды
- Балансировка потоков работы

### Flow Visualization
Визуализация потока работы:
- Отображение задач на доске
- Цветовая дифференциация типов задач
- Индикация приоритетов
- Отображение блокеров и рисков
- Визуализация зависимостей

### Lead Time & Cycle Time Analysis
Анализ времени выполнения:
- Измерение времени от создания до выполнения
- Анализ распределения времени
- Выявление узких мест
- Прогнозирование сроков выполнения
- Оптимизация процессов

## Интеграция с System Analyst

- Учет аналитических задач в потоке
- Визуализация задач анализа на доске
- Управление загрузкой System Analyst
- Отслеживание времени на аналитические работы
- Интеграция результатов анализа в поток разработки

## Примеры использования

### Пример 1: Настройка Kanban-доски для команды разработки
Колонки:
- Backlog: задачи для будущей работы
- Ready: задачи готовы к работе
- Analysis: анализ требований (2 WIP-лимит)
- Development: разработка (4 WIP-лимит)
- Testing: тестирование (2 WIP-лимит)
- Done: выполненные задачи

WIP-лимиты:
- Общий лимит: 8 задач
- По ролям: 2 аналитика, 4 разработчика, 2 тестировщика
- Мониторинг: ежедневно на standup-встречах

Временные метрики:
- Средний Lead Time: 8 дней
- Средний Cycle Time: 5 дней
- 80% задач выполняются за 10 дней

### Пример 2: Оптимизация потока задач
Проблема:
- Задачи застревают в колонке "Testing"
- Средний Cycle Time увеличился на 40%

Анализ:
- Lead Time: 8 → 12 дней
- Cycle Time: 5 → 7 дней
- Блокеры: нехватка тестировщиков, сложные интеграции

Решения:
- Увеличение WIP-лимита в Testing до 3
- Добавление автоматических тестов
- Параллельное тестирование отдельных компонентов
- Результат: Cycle Time уменьшился до 6 дней

### Пример 3: Управление различными типами задач
Классификация задач:
- Feature: новые функции (красный цвет)
- Bug: исправление ошибок (оранжевый)
- Tech Debt: техдолг (синий)
- Analysis: аналитические задачи (фиолетовый)

Политики:
- Bug-репорты имеют высокий приоритет
- Feature-задачи ограничены 60% от общего потока
- Tech Debt: не менее 20% времени
- Analysis: до 30% времени в зависимости от проекта
